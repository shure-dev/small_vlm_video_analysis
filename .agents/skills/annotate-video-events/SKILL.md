# 動画から単一イベントをアノテーションする

1本ずつ処理し、各動画から画面上で一度だけ起きる代表イベントを1種類選び、そのイベントへ区間を1つだけ付ける。反復イベントや瞬間的すぎる操作を無理に採用しない。既存の人手例は記述粒度の校正にだけ使う。

## 必要な入力を確認する

- 対象動画または順序付きフレーム
- フレーム時刻を秒へ変換する情報（fps、PTS、または時刻表）
- 同じ用途で作られた既存の人手アノテーション。なければ目的と希望粒度
- 必要な出力形式。指定がなければ後述の最小形式

媒体を読めない、時刻へ変換できない、または出力先が破壊的に上書きされる場合だけ停止して確認する。既存の人手アノテーションは、明示的な修正依頼がない限り変更しない。

## 1. 参考例から粒度を校正する

対象を見る前に、既存の人手例から代表的な5〜10動画を読む。次を短い校正メモにする。

- 一連の動作をどこからどこまで一つの作業単位とするか
- 対象物、道具、手段、方向、置き場所をどこまで書くか
- 「手を伸ばす」など準備動作を含めるか
- 開始と終了を映像から判定できる表現になっているか

参考例のイベント数や区間数はコピーしない。「1動画・1イベント・1区間」の出力契約を優先する。映像上の区別と後段のTemporal Groundingに必要な具体性を残し、意図、品質、成功、危険性など画面から確認できない状態は書かない。

## 2. 対象を1本選び、媒体を固定する

未注釈の対象から1本だけ選ぶ。使用する動画hashまたはフレーム集合、時刻原点、annotation fpsを記録する。元動画は変更しない。

高fps動画では、最初に一定間隔のannotation framesを決定論的に作る。粗い走査の後、候補区間内はより細かいfpsまたは元フレームで確認する。少なくとも最終的に採用する区間の境界周辺は、利用可能な全フレームを順番に見る。

## 3. 視覚確認を分担する

サブエージェントを利用できる場合、次の2つを独立に実行する。サブエージェントにはファイルを書かせず、観察結果だけを返させる。最終書き込みはコーディネータだけが行う。

### 粗い走査

動画全体を先頭から末尾まで疎なフレームまたはコンタクトシートで確認し、候補イベントと概略区間を列挙する。この段階では一つに決めない。

```text
対象動画と参考アノテーションの記述粒度だけを使ってください。
動画全体を時系列に走査し、画面で確認できる動作候補を列挙してください。
各候補について、概略区間、対象物、手段、動画内の出現回数、境界の明瞭さ、
動画長に対する持続割合、annotation frame数を返してください。意図を推測せず、
ファイルは編集しないでください。
```

### 全フレーム・境界精査

粗い走査の結果を正解とは見なさず、対象の全annotation frameを順番に確認する。候補区間の前後は細かく見て、独立に開始・終了を提案する。

```text
対象動画、時刻表、粒度の参考例を使ってください。
全annotation frameを時系列に確認し、候補ごとに、同じ動作が動画内で一度だけか、
開始・継続・終了を複数フレームで確認できるか、境界前後で別状態へ移るかを
独立に判定してください。根拠フレームと時刻を示し、ファイルは編集しないでください。
```

サブエージェントを利用できない場合は、同じ情報を見ない独立した2パスとして順番に実行し、その制約を報告する。

## 4. 一つの代表イベントを選ぶ

次をすべて満たす候補だけを適格とする。

1. 動画全体で一度だけ起き、同じ意味の離れた再出現がない
2. 一つの連続したhalf-open intervalで表せる
3. 瞬間ではなく、開始・継続・終了を複数フレームで確認できる
4. 時間解像度に対して十分長く、境界の小さな誤差だけでtIoUが極端に変わらない。別指定がなければ動画長の10%以上、かつ6 annotation frames以上を必須とする
5. 細かな接触や一瞬の操作へ分割しすぎず、一つの視覚的にまとまった作業段階である
6. 動画全体を覆う曖昧な要約ではなく、開始と終了を映像から説明できる
7. 他候補と対象物、手段、状態遷移で区別できる

適格候補が複数ある場合は、境界の明瞭さ、区間内での動作の一貫性、十分な長さ、作業内容の代表性の順で一つだけ選ぶ。長さ条件は候補を作るために緩めず、対象ドメインや評価設計から別の閾値が明示された場合だけ置き換える。

反復する候補を一つの長い区間で囲ってはいけない。反復の一部だけを恣意的に選んでもいけない。適格候補がない動画は、無理に正解を作らず `skip: no_single_sustained_event` と別のレビュー記録へ残す。

## 5. イベント文と一つの区間を確定する

イベント文は「誰が＋何を＋どうする」を基本にし、一つの候補を識別するために必要な対象物、道具、手段、方向、置き場所を加える。一連の開始から終了までを同じ文で説明できる粒度にする。

区間はhalf-open interval `[start_s, end_s)`にする。

- `start_s`: 定義した動作が最初に視認できる時刻
- `end_s`: 定義した動作が終了した後の最初の時刻
- 準備や後片付けは、イベント文に含む場合だけ区間へ含める
- クリップ端で打ち切られ、開始または終了を観測できない候補は原則として選ばない
- モデル予測へ合わせてイベント文や境界を動かさない

粗い走査と境界精査が1 annotation frameを超えて食い違う場合は該当フレームを再確認する。解消できない候補は不採用にし、次順位の候補を選ぶ。

## 6. 独立レビューを行う

可能なら別のサブエージェントへ、対象動画、校正メモ、完成候補だけを渡す。モデル予測や期待スコアは渡さない。次を確認させる。

- event IDと区間がそれぞれ一つだけである
- 同じ意味の動作が動画内の別時点に再出現しない
- 既定または明示された長さ条件を満たし、開始・継続・終了を複数フレームで確認できる
- 細かな操作へ分割しすぎず、動画全体の曖昧な要約にもなっていない
- イベント文が区間内の動作と手段を正確に表している
- 境界の外側に同じ動作が続いていない

レビュー指摘は証拠フレームを自分で確認してから反映する。エージェントが作成したイベント文と区間は、独立レビューを通るまではGT候補であり、確定した事実として扱わない。

## 7. 保存して検証する

指定schemaがなければ、意味と区間を分離した次の最小形式を使う。

```json
{
  "video_id": "clip_001",
  "interval_convention": "half-open_seconds",
  "event_labels": {
    "event_001": "観測可能な一連の動作を表すイベント文"
  },
  "events": {
    "event_001": [
      {"start_s": 2.0, "end_s": 7.0}
    ]
  }
}
```

保存後、`event_labels`と`events`のキーが同じ一つのevent IDだけであること、そのevent IDに区間が一つだけあること、`0 <= start_s < end_s <= duration`、schema外fieldがないことを機械的に検証する。証拠やskip理由は正解schemaへ混ぜず、別のレビュー記録へ置く。

## 8. VLM推論とtIoU改善を反復する

推論環境がある場合、人手正解の品質確認とは分離したうえで、指定された動画VLM（例: Marlin-2B）を複数回実行する。

1. 人手annotationをrevisionまたはhashで固定する
2. event IDを保ち、人手イベント文からモデル向けqueryを作る
3. GT時刻をモデルへ見せず、新しいrun IDで推論する
4. raw出力、query、モデルrevision、入力動画hashを保存する
5. event別と全体のtIoUを計算する
6. 不一致の原因をquery、モデル能力、GT候補の誤りに分けて調べる
7. queryが原因なら、GTを固定したままqueryだけを改善して新しいrunで再実行する
8. GT候補が疑わしい場合は、予測を隠した独立再監査を行う

GTとpredictionを同じファイルへ保存しない。完了済みrunを上書きしない。同じ動画でqueryを調整したスコアはdevelopment値であり、未見性能として報告しない。

### 低tIoUの原因を分ける

| 症状 | 最初に確認すること | 対応 |
|---|---|---|
| 別の動作を検出 | 対象物、道具、手段が曖昧 | 映像と過去例を見てqueryを具体化する |
| 動作の中心だけ検出 | queryが結果または一瞬の動作だけを表す | 開始から終了まで視認できる動作を記述する |
| 境界がずれる | 人手定義とqueryが異なるフェーズを指す | eventの開始・終了条件を言葉で揃える |
| モデルが別の一貫した動作を返す | queryの曖昧さ、またはGT候補のイベント文と映像の不一致 | query監査とGT再監査を分離して行う |
| 映像とイベント文・区間が不一致 | GT候補自体の誤り | predictionを見ずに映像証拠から別revisionとして修正する |

### GT候補を独立再監査する

VLMが明らかに別の内容や区間を返した場合、その出力を正解とは見なさない一方、GT候補も無条件に正しいとは仮定しない。不一致は再監査のきっかけとして扱う。tIoUが低いだけではGT誤りの証明にならず、高くてもGTの意味的正しさは保証されない。

次のいずれかならGT再監査を行う。

- モデルが映像上は成立している別の動作を一貫して検出する
- queryを具体化しても、イベント文と予測の不一致が解消しない
- 元の根拠フレーム、境界、イベントの一回性、長さ条件に不確実性がある
- イベント文が区間内の大半を説明していない可能性がある

再監査では次を守る。

1. 元のannotation revision、根拠、prediction runを固定して上書きしない
2. 再監査担当にはprediction区間、tIoU、期待する修正方向を見せない
3. 動画全体と全annotation frameを見直し、イベントの意味、一回性、長さ、開始、終了を独立に提案する
4. 独立案を作った後にだけ元GT候補と比較する
5. 元GT候補が映像証拠と不一致なら、根拠フレームと変更理由を残して新しいrevisionを作る
6. 元GT候補が妥当なら変更せず、queryまたはモデル能力の問題として記録する
7. 判断が割れる場合は境界を推測せず、`needs_review`またはskipとして扱う

モデルのprediction区間をGTへコピーしない。predictionへ近づける目的で境界やイベント文を変更しない。GT revisionを変更した場合、それ以前のスコア比較を中断し、新revisionを固定してbaselineから再実行する。

queryへ正解時刻や答えとなる秒数を入れない。tIoUを上げるためにGT境界をpredictionへ寄せない。

既定では最大3ラウンド、または事前に定めた目標（指定がなければmean tIoU `0.6`）まで反復する。改善が止まった場合はqueryを無限に変更せず、モデル能力または時間解像度の制約として記録する。最良runだけでなく全runと変更理由を残す。

## 完了条件

- 対象動画を先頭から末尾まで確認した
- 粗い走査と全フレーム精査を行った
- 一度だけ起きる、十分な長さの代表イベントを一つ選んだ
- event ID、イベント文、時間区間がそれぞれ一つだけである
- 反復、瞬間的すぎる操作、曖昧な全体要約を採用していない
- 独立レビューまたは独立した再確認パスを通した
- モデル出力を正解作成に使用していない
- schema、区間数、時刻範囲の検証に通った
- 適格候補がない場合は正解を捏造せずskipした
- 推論可能な場合、固定GTでbaselineと改善runを分けて評価した
- 低tIoUの原因をquery、モデル、GT候補に分け、判断理由を残した
- GT候補を修正した場合はpredictionを隠した再監査、新revision、baseline再実行を行った

英訳は人手正解を固定した後に推論queryとして作成し、元の人手イベント文を置き換えない。
